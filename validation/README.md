# validation/ -- independent cross-validation study

## Scope and honesty statement (read this first)

**This directory is deliberately OUTSIDE two rules that govern the rest of
the repository:**

1. **R9 (Python-3-standard-library-only tooling).** Every other Python
   tool in this repo (`tools/viz`, `tools/evidence`, `tools/compliance`,
   `tools/forensics`, `tools/prioritization`, `tools/docs`, `tools/readme`)
   is stdlib-only by spec rule R9, specifically so the generation/reporting
   pipeline never depends on a third-party package version. `validation/`
   is the **one exception**: it exists specifically to compare ADSC's
   models against established THIRD-PARTY implementations (an independent
   two-body propagator, an independent atmosphere model), which is
   impossible without third-party packages. See `requirements.txt` for the
   exact pinned versions used.
2. **The byte-identical reproducibility gate** (the CI job described in
   `.github/workflows/ci.yml` as "reproducibility gate (single authority)"
   and in `docs/regenerate_all.sh`). That gate certifies that ADSC's own
   C++ build regenerates `generated/*.csv` byte-for-byte from a clean
   clone. **`validation/` is not part of that pipeline, is not run by CI,
   and its scripts are not invoked by `regenerate_all.sh`.** Its outputs
   (`validation/*_results.csv`, `validation/decay_part*.csv`) are committed
   for transparency but are not regenerated or checked by any CI job, and
   are not claimed to be byte-reproducible across machines (they depend on
   `scipy`/`hapsira`/`pymsis` internals, which are not pinned as tightly
   as the C++ toolchain the reproducibility gate controls for).

**What this study is:** a one-time, honestly-reported cross-validation of
three ADSC model components (the CW/Hill relative-motion closed form, the
Vallado exponential-atmosphere decay model, and the WP13 EDT inclination-
efficiency formula) against independent implementations, run once (2026-07)
with pinned dependency versions, to address the citability-gap-analysis
finding that ADSC's numbers were "reproducible from a clean clone" but
never checked against anything external
(`_tasks_local/citability-gap-analysis.md`, section "#5 -- Independent
cross-validation"). See `RESULTS.md` for the actual numbers.

**What this study is NOT:**
- It is NOT a new, permanently-maintained validation suite. It is not
  re-run on every commit, and nothing elsewhere in the repo depends on its
  output being current. If `src/decay.cpp`, `src/relmotion.cpp`, or the
  WP13 EDT formulas change, these scripts should be re-run and
  `RESULTS.md` updated by hand -- there is no CI hook enforcing that.
- It does NOT "validate" ADSC in the sense of proving the models are
  correct or fit for any operational purpose. Every script's docstring and
  `RESULTS.md` state explicitly what each check does and does NOT cover
  (e.g. the CW check covers linearization error only, not J2/drag/attitude/
  control; the decay check covers the atmosphere-density assumption, not
  orbit-averaging or solar-activity forecasting; STELA, a real operational
  lifetime tool, was not run at all -- see `RESULTS.md`).
- Numbers here are reported as "consistent within X" or "differs by X",
  never as "validated" or "proven correct" (R7-style honesty discipline,
  applied here even though this directory is outside the spec's own rule
  numbering).

## Directory contents

| file | purpose |
|---|---|
| `requirements.txt` | pinned third-party package versions (Python 3.11.9, Windows) |
| `check_cw.py` | CW/Hill closed form vs full nonlinear two-body propagation |
| `check_decay.py` | Vallado exponential atmosphere vs NRLMSISE-00 (`pymsis`) |
| `check_edt_eta.py` | WP13 EDT eta(i) formula vs independent dipole-field quadrature |
| `RESULTS.md` | the actual numbers, agreement/disagreement tables, interpretation |
| `cw_results.csv`, `decay_part1_integrator_check.csv`, `decay_part2_atmosphere_check.csv`, `decay_part2_reference_point.csv`, `edt_eta_results.csv` | raw output of the three scripts (committed for transparency, not regenerated by CI) |

## How to re-run

```
cd validation
python -m venv .venv
# Windows:
.venv\Scripts\activate
# macOS/Linux:
# source .venv/bin/activate
pip install -r requirements.txt
python check_cw.py
python check_decay.py
python check_edt_eta.py
```

Each script is standalone, prints a human-readable report to stdout, and
writes its own CSV(s) alongside itself. None of them take command-line
arguments or read from outside the repository except
`generated/wp3_decay_trade.csv` (read-only, for the Part-1 integrator
cross-check in `check_decay.py`) and the header files under `include/adsc/`
and the derivation doc `wp13-edt-derivation.md` (read as reference numbers
to compare against in code comments; not imported/executed).

No internet access is required at run time: `pymsis`'s historical
F10.7/Ap index lookup is deliberately bypassed by passing explicit
`f107s=`/`f107as=`/`aps=` values to every `pymsis.msis.calculate()` call
(see `check_decay.py`), so nothing is downloaded. `hapsira`'s two-body
propagator similarly needs no ephemeris/network access for a pure
point-mass Cowell propagation. All three scripts are deterministic (no
RNG; the WP1/forensic-14/EDT cases use fixed, documented initial states).

## What was tried and did not work (honesty log)

- **`pyatmos`** (the task's first-choice atmosphere package, offering
  NRLMSISE-00/COESA76) installs, but (a) its `coesa76` module imports
  `pkg_resources`, which recent `setuptools` (>=81, installed here as
  82.0.1 by default) no longer ships without a compatibility warning/pin,
  and (b) importing the top-level `pyatmos` package unconditionally
  attempts to download an IERS Earth-orientation-parameter file over
  HTTPS at import time, which failed in this environment with
  `SSL: CERTIFICATE_VERIFY_FAILED`. Both are fixable (pin
  `setuptools<81`; pre-seed/mock the IERS cache or patch around the
  download), but they make `pyatmos` a poor fit for a "install pinned
  deps, run deterministically, no network at runtime" study. **We used
  `pymsis` instead** (github.com/SWxTREC/pymsis, the University of
  Colorado LASP / Space Weather Technology, Research and Education Center
  wrapper around the same public-domain NRLMSISE-00 Fortran source, plus
  the newer MSIS2/2.1 models) -- it imports cleanly, needs no network
  when explicit solar-activity indices are supplied, and directly exposes
  `version=0` for classic NRLMSISE-00 (used here) alongside MSIS2.1 (used
  only for a robustness spot-check, see `RESULTS.md`).
- **`hapsira`** (first-choice two-body/CW package) installs but its
  pinned/unpinned `astropy` dependency resolved to 7.2.0 at install time,
  which no longer exports `astropy.coordinates.matrix_utilities.
  matrix_product` that hapsira 0.18.0 imports at module load. Fixed by
  pinning `astropy==6.1.7` (see `requirements.txt`); `poliastro` itself
  (hapsira's now-archived upstream) was not additionally tried once
  hapsira was working, per the task's "try hapsira, else poliastro"
  ordering.
- STELA (CNES) was not attempted at all: it is a desktop Java Swing GUI
  application, not a scriptable library, and is out of scope for a
  pinned, one-time, non-interactive Python study. See `RESULTS.md` for
  what STELA would add beyond `check_decay.py`.
